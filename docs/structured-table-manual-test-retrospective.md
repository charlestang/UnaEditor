# 结构化表格 Manual Test 复盘

## 目的

这份文档不是 changelog，也不是功能说明。它的目标是把本轮 `add-structured-table-editing` 手工验证中反复暴露出来的问题，收敛成几条**真正决定成败的架构理解**。

如果下次还要继续做这类“CodeMirror + replace widget + Vim + overlay 编辑桥”的重交互功能，提示词里只要一开始就把这些关键原理点明，很多弯路可以直接避开。

---

## 一句话总结

这轮问题并不是零散的交互 bug 堆积，而是反复落在下面几层理解上：

1. **谁是光标和位置的唯一真相源**
2. **哪些 DOM 属于 CodeMirror 正文，哪些只是 widget UI**
3. **真实用户事件链，和测试里简化触发路径，不是一回事**
4. **结构操作的“保护”和“整表收口”必须分开理解**
5. **sandbox 验证入口不能假设自己能绕开编辑器真实输入链**

只要这五条一开始没想清楚，后面就会不断出现：

- 看起来像焦点问题，其实是 selection 真相源错了
- 看起来像菜单点击失效，其实是事件链在更早阶段就被吞了
- 看起来像某个按钮无效，其实是验证入口走错了路径
- 看起来像边界 case，其实是结构删除规则定义本身就错了

---

## 本轮问题集中在哪些架构层

### 1. 光标系统层

最核心的问题，不是“cell 能不能编辑”，而是：

> **表格内当前位置，到底由谁表达？**

如果这件事不先定清楚，所有导航、Vim、编辑态切换、滚动跟随都会变得拧巴。

本轮最终收敛出的正确理解是：

- **CodeMirror 主 selection 是唯一真相源**
- `focusCell` 只是派生缓存，不是权威位置
- normal mode 下的光标显示，必须走 CodeMirror / Vim cursor layer
- overlay 只能做 insert/editing 输入桥，不能承担 normal-mode caret 语义

这条一开始没点明，就会不断出现下面这类误判：

- “active cell 有了，为什么没有真正的 caret”
- “点击 cell 以后像选中了，但不能像文本一样导航”
- “Vim normal mode 进入表格后为什么感觉不像在文档里”

### 2. Widget / DOM 所属层

表格是 `Decoration.replace({ block: true, widget })`。这意味着：

- 它**视觉上**在文档里
- 但它**不是普通正文 DOM**

如果没有明确区分“正文 DOM”和“widget UI DOM”，CodeMirror 会错误地试图把 handle、按钮、wrapper 子节点映射回正文位置。

本轮典型症状：

- `Invalid child in posBefore`
- 点 handle 后 selection 莫名跳到表格边界
- 右键菜单逻辑看起来没问题，但正文映射被 DOMObserver 干扰

最终收敛出的关键理解是：

- widget wrapper 应明确 `contenteditable="false"`
- 这类 widget UI 不应被当作正文可编辑 DOM
- `ignoreEvent()` / 事件路由必须结合 widget 角色来设计

### 3. 事件链层

本轮一个非常大的坑是：

> **自动化里“触发了某个事件”不等于真实用户路径是通的。**

典型例子：

- 测试里直接 `trigger('contextmenu')` 再点 `menu-delete`，可以过
- 真实用户路径却是：
  1. 左键点 handle
  2. 右键打开菜单
  3. 再点删除

这中间只要某个更早阶段的 `mousedown` 把事件吞掉，后面的 `click/contextmenu` 根本不会发生。

本轮最终暴露出的关键事实：

- 不能在整个 table wrapper 的 `mousedown` 上一刀切 `preventDefault`
- “cell 点击进入编辑”这条链可以抢在 `mousedown`
- 但 handle / menu / append button 的链路必须保留后续事件
- 菜单动作本身在复杂环境下用 `mousedown` 往往比 `click` 稳定

### 4. 结构操作语义层

有一类问题表面像 bug，实际上是 spec 本身写错了。

本轮典型有两个：

#### `Backspace` 删除最后一行空行

最初定义是：

- “最左首位 + 最后一行 + 整行为空”时，`Backspace` 删整行

最后确认这条没有产品价值，应该删除。

原因：

- 这是隐蔽结构操作，不符合用户心智
- 已经有更明确的结构入口：handle / 菜单 / `dd`
- 会抬高状态复杂度，但收益很小

#### 删除最后一列

最初定义是：

- “最后一列保护，不允许再删”

最后确认这条也不对。

更合理的规则是：

- 表头保护：防误删结构骨架
- 最后一列删除：**删除整张表**

也就是说：

> “保护”与“整表收口”是两件不同的事，不能混成同一种保守规则。

### 5. 验证入口层

这轮还反复暴露了一个事实：

> **sandbox 的验证入口，不能假设自己可以随便绕开编辑器的真实输入链。**

最典型的就是撤销：

- 真键盘 `Cmd+Z` 有时是通的
- sandbox 按钮却不通

最后发现，问题不在“撤销逻辑”，而在：

- 编辑器本体是否接上 `historyKeymap`
- 按钮是不是复用了真实键盘那条事件链
- 当前真实输入目标到底是 overlay 还是 `contentDOM`

这类问题如果不先区分：

- “功能本体坏了”
- “验证入口走错了”

很容易一直在错误层面修补。

## 本轮最重要的五条关键理解

### 1. 主 selection 是唯一真相源

提示词里最好直接点名：

> 表格内的当前位置必须由 CodeMirror 主 selection 表达；`focusCell` 只能是派生缓存；overlay 不能成为 normal mode 光标的真相源。

这是整个表格 / Vim 体验能不能自然成立的前提。

### 2. replace widget 不等于正文 DOM

提示词里最好直接点名：

> 结构化表格是 replace widget；其 wrapper、handle、menu、按钮都是 widget UI，不应被 CodeMirror 当作正文 DOM 去做位置映射。

否则就会不断冒出：

- DOMObserver 报错
- selection 奇怪跳转
- 事件命中和正文映射相互污染

### 3. 区分“真实用户路径”和“测试简化路径”

提示词里最好直接点名：

> 不要只验证简化触发路径；必须优先验证真实用户链路，例如：mousedown -> click/contextmenu -> menu action，而不是只在测试里直接触发最终按钮。

这条会极大减少“自动化绿了，真实还是坏的”。

### 4. 结构删除要区分“保护”和“收口”

提示词里最好直接点名：

> 表头保护属于防误删；删除最后一列、仅剩表头时删除等边界属于整表收口，不应一概用“最后一个不能删”的保守规则处理。

这条可以提前避免很多错误 spec。

### 5. 验证按钮必须复用真实输入链

提示词里最好直接点名：

> sandbox 里的验证控制项不是“实现一套新逻辑”，而是“稳定触发当前编辑器实例上真实会生效的那条输入路径”。

尤其是：

- undo / redo
- context menu action
- overlay 焦点下的快捷键

## 哪些问题看起来很多，其实是同一个根

下面这些问题表面分散，实际上经常是同一个根因在不同层的投影：

### A. “看起来是焦点问题”

常见表现：

- 点击表格外后，光标没出现
- `Cmd+Z` 被浏览器接管
- normal mode 进入 cell 后没有真正的 caret

背后往往不是单纯 blur/focus，而是：

- 主 selection 没收好
- 当前真实输入目标判断错了
- overlay 还在抢输入链

### B. “看起来是某个按钮/菜单没反应”

常见表现：

- 菜单能弹，但删不掉
- 按钮按下去无效
- 自动化里能点，真实页面里不能点

背后往往不是业务逻辑没写，而是：

- `mousedown` 太早吞事件
- `click` 根本没到
- 菜单项在焦点切换中被提前清理

### C. “看起来是某个边界 case 没考虑”

常见表现：

- 删除最后一列
- 仅剩表头时 `dd`
- `Backspace` 删空行

背后很多时候不是实现疏漏，而是：

- 规则本身就写得不对
- 保护和收口没有分开

## 下次写提示词时，建议直接点名的内容

下面这段可以直接当提示词骨架复用。

### 提示词骨架

> 这是一个 CodeMirror 6 的 replace-widget 表格编辑问题。请先按下面原则理解再动手：
>
> 1. 表格内位置的唯一真相源是 CodeMirror 主 selection，`focusCell` 只是派生缓存。
> 2. 表格 widget 的 wrapper / handle / menu / add button 都是 widget UI，不是正文 DOM；不要让 CodeMirror 把它们当作可编辑正文节点。
> 3. overlay 只承担 insert/editing 输入桥，不承担 normal mode caret 语义。
> 4. 请优先验证真实用户事件链，而不是只依赖测试里的简化触发路径。尤其关注 `mousedown -> click/contextmenu -> menu action` 是否完整发生。
> 5. 如果问题涉及结构删除，请区分“保护规则”和“整表收口规则”，不要默认用“最后一个不能删”的保守策略。
> 6. 如果问题涉及 sandbox 按钮或验证入口，请确保它复用真实输入链，而不是绕开编辑器状态机直接调用底层命令。
>
> 在开始修复前，请先回答：
> - 这个问题属于光标真相源、widget DOM 归属、事件链、结构规则，还是验证入口问题？
> - 当前真实用户路径是什么？
> - 测试路径和真实路径是否一致？

### 如果是 Vim / normal mode 相关

可以再追加一句：

> 请默认把 Vim normal mode 的 caret 当作 CodeMirror / `.cm-vimCursorLayer` 的职责，而不是浏览器原生输入框 caret。

### 如果是结构菜单相关

可以再追加一句：

> 请不要只验证“直接触发菜单项 click”；必须验证用户实际的右键链路是否成立。

### 如果是撤销 / 重做相关

可以再追加一句：

> 请先分清是撤销栈坏了，还是触发路径错了；不要一开始就直接从外部调用 undo 命令。

## 建议的排查顺序

以后遇到同类问题，建议固定按这个顺序排查：

1. **先定真相源**
   - 当前光标/选区到底由谁表达？

2. **再看真实用户路径**
   - 用户到底是点了什么、按了什么、触发了什么顺序？

3. **再看 widget DOM 归属**
   - 哪些节点是正文，哪些只是 UI？

4. **再看 spec 是否写对**
   - 这是实现错了，还是规则本身就不合理？

5. **最后才补测试和 sandbox 入口**
   - 不要把验证入口当成功能本体

## 对这轮工作的最终判断

这轮最大的收获，不是修完了多少个具体 bug，而是把下面这些误区逐步排除了：

- “表格编辑问题主要是样式和焦点问题”
- “菜单能弹出来说明链路就没问题”
- “自动化测得通，真实路径大概率也通”
- “越保守越安全，最后一列/最后一行最好都保护住”
- “sandbox 按钮可以随便直接调用底层能力”

最终真正有效的理解是：

> 结构化表格不是“普通 DOM 上叠点交互”，而是一个必须和 CodeMirror selection、widget 边界、真实事件链、结构语义一起设计的系统。

这条理解一旦先立住，后续很多问题都会从“撞运气修 bug”变成“按层定位并收敛”。
