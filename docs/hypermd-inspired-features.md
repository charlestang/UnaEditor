# HyperMD 借鉴功能需求草案

本文档记录 `una-editor` 未来可以参考的 5 个功能方向，仅作为需求草案与备忘，不代表当前版本承诺实现，也不代表已经进入开发排期。

如果未来决定正式推进，建议再为单个功能创建对应的 OpenSpec change，把这里的内容细化为 proposal / design / specs / tasks。

## 1. 脚注系统

### 功能目标

让编辑器把 Markdown 脚注视为一等内容对象，而不是普通文本。用户在编辑态中既能保留原始 Markdown 源码，又能获得脚注引用、跳转和预览能力。

### 基本需求

- 编辑器应能识别常见 Markdown 脚注引用与脚注定义语法。
- 在 `livePreview` 模式下，非激活状态的脚注引用应具备更接近最终阅读态的视觉表现。
- 当光标进入脚注引用或定义区域时，编辑器应恢复原始 Markdown 源码显示，保证直接编辑路径可用。
- 用户应能从脚注引用跳转到对应脚注定义。
- 用户从脚注引用跳转后，系统应提供返回原位置的能力。
- 编辑器应支持在悬浮或其他轻量交互中预览脚注内容。
- 如果脚注引用未找到对应定义，系统应提供保守降级行为，不得破坏原始文本。
- 若未来引入相对链接解析能力，脚注中的链接地址应能复用同一套解析逻辑。

### 暂不包含

- 富文本脚注编辑器。
- 脚注自动编号重排。
- 脚注拖拽排序或批量管理面板。

## 2. 可插拔代码块 Renderer 体系

### 功能目标

在现有代码块 live preview 基础上，允许特定语言标识的 fenced code block 渲染为图表、流程图或其它自定义内容，同时保持 Markdown 源码仍然是唯一真相。

### 基本需求

- 系统应允许基于 fenced code block 的语言标识注册自定义 renderer。
- 当某个代码块命中 renderer 时，编辑器应在非激活状态下显示渲染结果，而不是只显示普通代码文本。
- 当光标进入该代码块任意位置时，编辑器应恢复原始 fence 源码与正文代码内容。
- 如果某个代码块未命中 renderer，系统应回退到普通代码块 live preview 行为。
- 渲染结果应与现有代码块壳子、标题行、语言标签、复制按钮等体验保持兼容，避免出现两套完全割裂的 UI。
- 系统应允许宿主侧注册、替换或禁用某个 renderer。
- 渲染失败时，编辑器应提供安全降级，不得导致代码块内容不可见或不可编辑。
- 首期可优先面向 `mermaid` 之类高价值场景设计接口，但接口本身不应只绑定单一渲染器。

### 暂不包含

- 在线执行任意代码。
- 复杂沙盒运行时。
- 与外部服务强绑定的专有 renderer。

## 3. HTML 粘贴转 Markdown

### 功能目标

提升从网页、文档工具、知识库或富文本编辑器复制内容到 `una-editor` 时的体验，让系统尽可能保留结构语义，并自动转换为合法 Markdown。

### 基本需求

- 当用户粘贴内容同时包含 `text/html` 与 `text/plain` 时，系统应支持优先尝试将 HTML 转换为 Markdown。
- 转换能力应覆盖高频结构，包括标题、段落、强调、粗体、链接、图片、列表、引用、代码块和表格。
- 当 HTML 中不存在值得转换的结构时，系统可回退到普通纯文本粘贴路径。
- 转换后的内容应尽量符合当前编辑器支持的 Markdown 语法范围，避免生成不稳定或不可维护的输出。
- 对于不支持或不安全的 HTML 片段，系统应采用保守降级策略，而不是盲目保留原始 HTML。
- 转换逻辑应允许未来按需扩展 GFM 表格、删除线等规则。
- 系统应允许宿主侧选择关闭 HTML 转 Markdown，回退为纯文本粘贴。
- 该能力不应干扰现有结构化表格 overlay、普通纯文本粘贴和图片文件粘贴路径。

### 暂不包含

- 完整 HTML 白名单渲染引擎。
- 样式级别的高保真还原。
- 富文本源应用的 1:1 排版复刻。

## 4. 文件插入 / 上传流水线

### 功能目标

把当前“只抛出图片文件数组”的能力提升为完整文件插入流水线，让宿主能够在粘贴或拖拽文件时获得统一的占位、上传、回写与失败处理机制。

### 基本需求

- 编辑器应支持通过粘贴或拖拽接收文件输入。
- 文件输入不应只限于图片，系统应为图片与通用文件保留可扩展入口。
- 当宿主接受文件处理请求后，编辑器应能在当前插入位置显示占位内容。
- 占位内容应支持处理中、成功、失败等基础状态。
- 当宿主完成上传或处理后，系统应能把占位内容替换为最终 Markdown 文本。
- 宿主应能控制最终插入文本的内容，例如图片 Markdown、附件链接或自定义语法。
- 文件处理流程应支持取消或失败回退，避免编辑器中残留不可恢复的坏状态。
- 如果宿主未提供处理器，编辑器应保守回退，不强行吞掉默认行为。
- 这套机制应与现有 `drop(files)` 事件兼容或可平滑迁移，避免破坏已有集成。

### 暂不包含

- 内建云存储服务。
- 内建上传 UI 平台。
- 多文件上传队列管理面板。

## 5. 数学公式 Live Preview

### 功能目标

在 Markdown 编辑态中为行内公式和块级公式提供即时预览，使技术写作、学术笔记和工程文档场景更完整。

### 基本需求

- 编辑器应支持识别常见的行内数学公式与块级数学公式语法。
- 在 `livePreview` 模式下，非激活状态的数学公式应显示为渲染后的公式结果。
- 当光标进入公式范围时，编辑器应恢复原始公式源码，允许用户直接编辑。
- 系统应支持为公式渲染器提供可替换适配层，例如未来接入 KaTeX 或 MathJax。
- 公式渲染失败时，系统应给出保守降级行为，例如回退源码或显示错误态，而不是直接吞掉内容。
- 公式块应与当前内容版心、滚动、选区和主题体系兼容。
- 数学公式能力不应破坏普通美元符号输入或其它已有 Markdown 行为。
- 首期可以先聚焦只读渲染与源码回退，不要求提供复杂公式工具栏。

### 暂不包含

- 可视化公式编辑器。
- 公式自动补全。
- 宏管理、编号引用、交叉引用等高级 LaTeX 工作流。

## 选型建议

如果未来只打算优先推进 1 到 2 个功能，我建议优先考虑：

- 脚注系统：与当前 Markdown 写作场景最贴近，也最容易和现有 AST / live preview 模型衔接。
- HTML 粘贴转 Markdown：对真实使用体验提升很直接，尤其适合内容迁移场景。

如果未来更想增强产品差异化，可以优先考虑：

- 可插拔代码块 Renderer 体系。
- 数学公式 Live Preview。
