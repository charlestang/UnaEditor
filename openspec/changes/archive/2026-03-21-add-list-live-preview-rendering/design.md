## Context

`UnaEditor` 当前的 `livePreview` 能力已经在 `src/extensions/hybridMarkdown.ts` 中为标题、强调、链接、图片、blockquote 和 fenced code block 提供了不同程度的渲染增强，但列表仍然停留在纯 Markdown 源码态。对于笔记、文档和知识库场景，列表是最常见的块级结构之一；列表缺失会让整个编辑体验在视觉上仍然明显偏向“源码编辑器”，与项目希望接近 Obsidian 的产品方向不一致。

本次变更不追求完整富文本列表编辑，而是在现有 hybrid 渲染架构上为标准列表提供保守增强：在非激活状态下渲染为接近最终阅读态的列表，在进入当前列表项时退回 Markdown 源码。范围同时覆盖 GFM task list，但明确不提供点击切换、拖拽排序或富交互 checkbox。

现有架构约束包括：

- Markdown 解析继续依赖 `@codemirror/lang-markdown`
- hybrid 渲染主要通过 syntax tree、line decoration、mark decoration 和 widget 完成
- `livePreview` 已经对光标进入结构后的源码回退、导航修正和视口内装饰更新建立了基本模式
- 现有 spec 已明确表格保持源码态，本次不扩大到 table

## Goals / Non-Goals

**Goals:**

- 在 `livePreview` 模式下支持 CommonMark 标准无序列表和有序列表的保守渲染
- 在 `livePreview` 模式下支持 GFM task list 的只读 checkbox 渲染
- 在光标进入当前列表项时恢复该列表项的 Markdown 源码，以保证直接编辑 marker 与 task 状态
- 保持嵌套列表、混合 marker 和现有 hybrid 渲染行为兼容
- 明确首页 Demo 与 sandbox 的产品职责边界，避免展示内容和测试内容混淆
- 不新增对外 props 或额外运行时依赖

**Non-Goals:**

- 不实现 Markdown table 的渲染
- 不提供 checkbox 点击切换、hover 工具条、拖拽排序或其他富交互列表控件
- 不引入 markdown-it、remark 等额外 Markdown 渲染引擎
- 不把整个列表整体替换为不可直接编辑的块级 widget
- 不把首页 Demo 扩展成完整功能控制台

## Decisions

### 1. 语法标准基线采用 CommonMark 列表 + GFM task list

**决策**：本次支持范围以 CommonMark 的标准列表语法和 GFM 的 task list 扩展为基线，而不是只支持 `-` 或 `1.` 这类单一写法。

覆盖的输入范围至少包括：

- 无序列表 marker：`-`、`*`、`+`
- 有序列表 marker：数字加 `.` 或 `)`
- task list marker：`[ ]`、`[x]`、`[X]`
- 上述写法在嵌套列表中的组合

**理由**：

- 这是一组最主流、最符合用户预期的 Markdown 列表写法
- 当前 CodeMirror Markdown 语法树已经能够识别标准列表 marker，无需引入第二套解析系统
- 先把“标准覆盖”写清楚，后续实现和测试边界才不会漂移

**替代方案**：

- 仅支持 `-` 与 `1.`：实现更小，但会人为制造与标准不一致的体验缺口
- 追求更广泛的方言兼容：覆盖更大，但当前并无足够需求支撑

### 2. 列表采用“保守渲染”而不是整块 widget 化

**决策**：列表在非激活状态下只做 marker 与 task 标记的视觉替换和必要的行样式增强，不把整个列表块替换成富文本 DOM 结构。

**理由**：

- 与当前 hybrid 架构一致，避免把列表变成第二套富文本系统
- 用户感知收益已经足够高：看到 bullet、ordered marker 和 checkbox，阅读态会明显更接近成品
- 可最大程度保留源码编辑路径，减少选区、坐标映射和事务同步复杂度

**替代方案**：

- 将列表整块替换为 HTML `ul` / `ol` / `li` 结构：阅读态更强，但编辑与定位成本显著上升
- 仅隐藏 Markdown marker 不做任何视觉替代：实现简单，但用户感知不足

### 3. 激活粒度采用“列表项”而不是“整个列表”或“仅当前字符”

**决策**：当光标或选区进入某个列表项时，该列表项恢复源码态；列表外的其他列表项继续保持渲染态。实现上应以 `ListItem` 为主要激活单元，而不是将整个 `BulletList` / `OrderedList` 作为一个大 scope。

**理由**：

- 这最接近用户的心智模型：“我正在编辑这一项，而不是整个列表”
- 可避免进入某一项后整组列表突然全部回源码的视觉跳变
- 比按单个字符或单个 marker 控制更稳定，能更自然地覆盖续行和复杂光标位置

**替代方案**：

- 以整个列表为激活范围：实现可能更简单，但体验明显过粗
- 仅在点击 marker 时回源码：与用户“进入当前项即编辑源码”的预期不符

### 4. task list 通过列表项首段文本识别，而不是等待专用 AST 节点

**决策**：task list 的识别以列表项首段内容是否匹配 GFM task marker 为准，而不要求底层 Markdown 语法树先提供独立的 task 节点。

**理由**：

- 当前 `@codemirror/lang-markdown` 对 task list 没有现成的专用节点可直接消费
- 在现有列表语法树基础上，对列表项首段做轻量识别足以满足本次需求
- 可以在不扩展解析链路的前提下完成 GFM task list 支持

**替代方案**：

- 修改或扩展底层 Markdown parser 以引入 task 节点：语义更纯，但本次代价过高
- 只支持普通列表，不支持 task list：与本次范围不符

### 5. checkbox 保持只读视觉元素，不提供直接交互

**决策**：task list 在非激活状态下显示为 checkbox，但 checkbox 仅作为视觉表达，不承担直接切换勾选状态的交互职责。

**理由**：

- 保持 Markdown-first：状态切换仍然通过编辑源码中的 `[ ]` / `[x]` 完成
- 避免在渲染层引入额外事务、状态同步和可访问性交互复杂度
- 与现有 hybrid 模式“渲染态便于阅读，源码态负责编辑”的原则一致

**替代方案**：

- 点击 checkbox 直接切换源码状态：交互更强，但会显著放大设计和测试成本

### 6. 首页 Demo 与 sandbox 采用不同的信息密度

**决策**：`/#/` 首页 Demo 作为展示位，保留少量高价值主题控制和精选语法内容，强调第一观感与渲染质量；`/#/sandbox/` 作为验证位，保留更完整的控制项和更丰富的语法示例，便于回归测试与自由试用。

**理由**：

- 首页的首要目标是吸引用户理解组件质量，而不是一次暴露所有可调项
- sandbox 更适合承载高密度选项和手工验证内容
- 把两者职责写清楚，能避免后续在内容组织和控件数量上不断摇摆

**替代方案**：

- 首页和 sandbox 共享同一套控件与示例：实现简单，但会让展示面和验证面都失焦

### 7. 首页展示主题必须带动外层容器同步变化

**决策**：首页 Demo 中的编辑器主题切换不仅影响编辑器内部，还必须让外层展示容器、边框和表面底色形成连贯的亮色 / 暗色展示语境。

**理由**：

- 仅切换编辑器内部配色会造成“亮色内容放在深色壳子里”的半切换状态
- 首页是产品展示位，主题切换的完成度本身就是体验的一部分

**替代方案**：

- 只切编辑器内部主题：实现简单，但视觉上不成立

## Risks / Trade-offs

- **[Risk] task list 依赖文本识别可能与特殊链接或非常规内容相互干扰** → 仅在合法 `ListItem` 的首段前缀位置识别 task marker，避免对普通段落启发式扫描
- **[Risk] 列表项进入激活态时出现范围过大或范围过小的切换** → 以 `ListItem` 为激活边界，并补充嵌套列表、续行、混合 marker 的测试
- **[Risk] ordered list 的视觉渲染与源码 delimiter 存在差异感知** → 保持源码在激活态立即可见，并将 delimiter 覆盖纳入测试范围
- **[Trade-off] 不提供 checkbox 点击切换会牺牲部分“所见即所得”感** → 用更明确的源码编辑路径换取架构稳定性与一致性
- **[Trade-off] 保守渲染不会让复杂列表看起来像完整 HTML 输出** → 本次优先优化高频阅读体验，不追求整块富文本重建
- **[Trade-off] 首页 Demo 控件减少后，可探索性下降** → 通过 sandbox 保留完整控制面，分别满足展示和验证需求

## Migration Plan

1. 在现有 `livePreview` 渲染层中补充列表与 task list 的识别和装饰能力，不修改对外 API
2. 在组件测试中增加标准列表、嵌套列表、task list、激活态回源码和点击 checkbox 不切换的场景
3. 在 Playground 中增加列表演示内容，便于人工验证与回归
4. 如实现中发现复杂列表交互不稳定，可在不影响普通模式的前提下回退到源码态行为

## Open Questions

- 对于包含多段落、内嵌代码块或其他复杂 block 的列表项，首期是否一律保守回源码，还是继续渲染其首段 marker
- ordered list 在非激活态是否需要尽量保留源码中的 delimiter 视觉差异（`.` 与 `)`），还是统一呈现为常规有序列表样式
