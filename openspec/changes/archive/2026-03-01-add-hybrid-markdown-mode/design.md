## Context

`UnaEditor` 当前基于 CodeMirror 6 提供标准 Markdown 源码编辑体验，核心扩展组合集中在 `src/composables/useEditor.ts`。编辑器已经具备主题、行号、只读、占位符、图片拖拽等基础能力，但尚未提供“阅读态接近最终内容、编辑时仍保留源码控制”的混合编辑体验。

本次变更要在现有编辑器架构中引入一个可开关的 hybrid Markdown 渲染模式，并继续使用 `@codemirror/lang-markdown` 作为 Markdown 解析基础。首期范围聚焦在：

- 行内 Markdown 结构的渲染态显示
- 图片语法的渲染
- blockquote 和 fenced code block 的保守视觉增强
- 保持表格为源码编辑态

该变更具有明显的性能和交互复杂度，因为渲染态会引入额外 decorations、widgets，以及基于光标位置的显示模式切换。

## Goals / Non-Goals

**Goals:**

- 为编辑器提供一个可启用或禁用的 hybrid Markdown 渲染模式
- 在不放弃 Markdown 源码编辑能力的前提下，提高文档的阅读态可读性
- 为行内语法和图片提供明显的“渲染态/源码态”切换体验
- 通过分层扩展设计，为未来增加更复杂的块级渲染（如表格）预留结构
- 将性能控制纳入首期设计，避免每次光标移动都触发高成本全量重算

**Non-Goals:**

- 首期不实现 Markdown 表格的完整 widget 渲染
- 首期不引入 Markdoc、MD AST 渲染器或额外的 Markdown 解析引擎
- 首期不追求完全所见即所得的块级替换体验
- 首期不处理图片上传、图片资源缓存策略或远程资源代理

## Decisions

### 1. 继续使用 `@codemirror/lang-markdown` 作为唯一解析基础

**决策**：保留 `@codemirror/lang-markdown` 作为 Markdown 语法树来源，不引入额外的解析引擎。hybrid 渲染仅基于 CodeMirror 现有语法树和装饰系统构建。

**理由**：

- 与当前编辑器架构天然兼容，不需要重建解析链路
- 继续依赖 CodeMirror 的语法树可避免维护双份文档模型
- 图片、标题、强调、blockquote、围栏代码块等首期目标都可以在现有 Markdown 语法能力内处理
- 首期已明确排除表格完整替换，因此无需为了 table 立即引入额外的渲染系统

**替代方案**：

- 引入 Markdoc/remark/markdown-it 做额外渲染：渲染能力更强，但会增加双模型同步成本和依赖复杂度
- 完全独立解析 Markdown 再反向映射到编辑器：灵活性高，但与 CodeMirror 选区和事务系统耦合成本过高

### 2. 将 hybrid 渲染拆分为“轻装饰层 + 重组件层”

**决策**：将 hybrid 渲染扩展拆分为两类：

- 轻装饰层：负责隐藏 Markdown 标记、增强行内结构视觉样式、提供 blockquote/code block 的保守视觉增强
- 重组件层：负责图片等高感知节点的 widget 渲染

**理由**：

- 行内装饰更新频率高，但成本较低，适合通过 decorations 处理
- 图片等节点会引入 DOM 结构和资源加载，必须与轻量装饰分离，避免所有变化都触发重型重建
- 为未来表格等更重的块级 widget 预留清晰的扩展落点

**替代方案**：

- 由单一插件统一处理所有渲染：实现集中，但性能调优和问题定位会更困难
- 首期全部使用 widget：用户感知强，但交互和性能风险过高

### 3. 图片作为首期唯一的“重渲染单元”

**决策**：首期图片语法作为唯一需要进入 widget 渲染路径的结构。图片在语法层仍视为 Markdown 原生语法节点，但在渲染层按“可替换渲染单元”处理。

**理由**：

- 图片的用户感知收益高，且明显优于单纯展示源码
- 相比表格，图片的源文本与渲染结果映射更简单
- 相比 blockquote 和代码块，图片更适合通过 widget 获得直观收益

**替代方案**：

- 首期不做图片渲染，只做文字类结构：实现更稳，但用户感知不足
- 首期同时做图片与多个块级 widget：功能更完整，但明显扩大首版风险

### 4. 使用“边界驱动 + 视口驱动”的更新策略

**决策**：hybrid 渲染更新不按“任何 `selectionSet` / `docChanged` 都全量重算”处理，而是以渲染单元边界和可见范围为核心。

具体原则：

- 光标移动时，仅在光标跨越某个渲染单元边界时切换渲染态/源码态
- 文档变更时，优先映射现有 decorations，并仅重算受影响范围及其邻近上下文
- 重组件层（图片 widget）优先限制在 `visibleRanges` 及其缓冲区附近处理

**理由**：

- 可以避免在每次光标移动时重建全部装饰或 widgets
- 对长文档更友好，成本随视口而不是随全文规模增长
- 更符合 CodeMirror 的事务和视口更新模型

**替代方案**：

- 每次 selection 或 doc 改变就全量扫描可见区域：实现简单，但滚动和光标移动时容易抖动
- 全文静态渲染：阅读态强，但编辑体验和性能都不稳定

### 5. Hybrid 模式对调用方显式暴露为可配置能力

**决策**：hybrid Markdown 渲染模式作为显式的组件配置能力暴露给调用方，而不是默认强制启用。

**理由**：

- 允许现有源码编辑场景保持原行为，避免对现有集成造成破坏
- 便于在 Playground、测试和未来业务侧分别验证普通模式与 hybrid 模式
- 为后续灰度启用或按场景启用提供空间

**替代方案**：

- 默认全量启用：用户能立即看到效果，但会引入行为变更风险
- 仅提供单独组件变体：功能隔离清晰，但维护成本更高

## Risks / Trade-offs

- **[Risk] 光标在渲染态和源码态之间切换时出现闪烁或跳变** → 仅在跨越渲染单元边界时切换，并优先让切换条件保持稳定
- **[Risk] 图片 widget 频繁销毁重建导致布局抖动** → 为图片 widget 引入稳定的等价判断，并限制在视口附近渲染
- **[Risk] blockquote / fenced code block 的“保守增强”视觉收益不足** → 首期明确接受这种折中，把重点放在稳定性和可编辑性上
- **[Risk] 未来要加入 table 时当前结构不足以承载** → 从首期开始就将轻装饰层和重组件层分离，为更重的块级 widget 留出架构位置
- **[Trade-off] 首期不做表格完整渲染会降低“所见即所得”程度** → 用更小的首期范围换取更稳的实现和更低的交互风险

## Migration Plan

1. 为编辑器增加 hybrid 模式配置入口，并保持默认关闭，以保证现有使用方不受影响
2. 在 Playground 中为 hybrid 模式增加显式演示和验证路径
3. 先落地轻装饰层与图片 widget 层，再补充 blockquote / fenced code block 的保守增强
4. 若发现交互或性能不稳定，可直接关闭 hybrid 模式回退到传统源码编辑态

## Open Questions

- 图片加载失败时，首期是否显示占位内容、alt 文本，还是直接退回源码态？
- 首期是否需要同时支持“独占一行的图片”与“段落内图片”的不同渲染策略，还是先统一处理？
- blockquote 和 fenced code block 的保守增强是否需要隐藏部分标记字符，还是只增强容器样式即可？
- 后续如果决定支持表格，是否沿用同一套重组件层，还是为表格单独设计更严格的交互模型？
