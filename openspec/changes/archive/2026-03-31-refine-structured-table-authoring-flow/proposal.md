## Why

当前结构化表格的 spec 已经定义了“合法表格进入结构化态以后”该如何编辑，但没有把“用户手动录入 Markdown 表格并触发激活”的 authoring flow 定义完整。结果是激活时缺少首个数据行、文档尾部表格缺少稳定的普通文本出口，以及用户删除尾随空行时会重新把自己推回语法脆弱区。

## What Changes

- 细化手动录入 Markdown 表格时的激活行为：当用户输入到合法 GFM 表格阈值并触发结构化激活时，编辑器自动补齐首个空数据行，并把光标放到该行第一列 cell 中继续录入。
- 细化文档尾部表格的 authoring continuity：当手动录入的表格恰好位于文档末尾时，编辑器继续自动在表格后保留一个普通文本空行，确保用户始终有可达的表格外编辑出口。
- 明确 `手动输入` 的判定边界：只覆盖从非表格态出发、由逐字符文本输入跨过合法 GFM 表格阈值的路径；完整表格的粘贴、撤销重放和程序化写入不应触发额外补行或尾随空行。
- 收紧首次激活的撤销语义：当最后一个输入字符触发表格激活，并伴随系统补齐首个空数据行或文档尾部空行时，第一次撤销应整体回退到触发临界点之前的源码状态。
- 新增文档末尾尾随空行的删除守卫：当用户位于该空行行首并执行后退删除时，第一次删除应进入上方整张表的暂态选中；第二次删除才真正删除整张表；若用户改为点击、方向键移动或继续输入，则退出该暂态并回到正常编辑路径。
- 补充自动化测试与手工验证要求，覆盖表格激活瞬间的首行补齐、文档尾部出口、激活撤销语义，以及两段式整表删除守卫的进入、退出和恢复。

## Capabilities

### New Capabilities

- None.

### Modified Capabilities

- `structured-table-editing`: 补充手动录入表格时的激活连续性、文档尾部表格的普通文本出口，以及尾随空行的两段式整表删除守卫。

## Impact

- 受影响代码主要集中在 `src/extensions/structuredTable.ts` 及相关表格源码改写、首次激活的 transaction filter 编排、焦点切换、键盘拦截、撤销边界和结构选中状态。
- 需要补充 `test/UnaEditor.test.ts` 中手动录入表格、粘贴完整表格、文档尾部激活、首次激活撤销，以及尾随空行删除守卫的回归测试。
- `/#/sandbox/` 的手工验证路径需要覆盖“手动输入表格激活”“文档末尾表格出口”“删除尾随空行时的整表选中退出/确认”“撤销恢复表格与尾随空行”这几条完整 flow。
- 对外不新增新的 API，但会改变 `livePreview=true` 时手动录入 Markdown 表格的交互语义。
