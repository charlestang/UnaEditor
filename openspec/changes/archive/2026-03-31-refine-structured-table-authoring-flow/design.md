## Context

当前 `structured-table-editing` 已经把结构化表格的稳态编辑语义定义得比较完整，但它默认假设“表格已经存在且用户已经进入表格内部”。这使得三个过渡态没有被统一设计：

1. 用户手动录入 `header + delimiter` 后，表格第一次从源码态切换到结构化态时，应该如何继续录入第一行内容。
2. 如果该表格正好处于文档末尾，系统如何同时满足“继续录表格内容”和“随时离开表格继续写普通文本”这两个目标。
3. 如果文档末尾那个辅助空行被用户删除，编辑器如何避免把换行直接并回表格末尾，导致表格语法坍塌或失去普通文本出口。

这次 change 的难点不在于再加一个表格功能，而在于把“手动录入 Markdown 表格”定义成一条连贯的 authoring flow，而不是几条互相打架的补丁规则。

## Goals / Non-Goals

**Goals:**

- 保持当前手动录入表格的激活阈值不变，仍然在表头行和 delimiter 行组成合法 GFM 表格时触发结构化激活。
- 在激活瞬间自动补齐首个空数据行，并把光标直接放到第一行第一列 cell，保证用户可以继续录入表格内容。
- 当该表格位于文档末尾时，同时保留表格后的普通文本空行，确保用户有稳定的表格外出口。
- 把文档末尾表格下方空行的删除改成两段式整表删除守卫，避免误删换行后破坏表格结构。
- 让这三条行为在同一套状态机里成立，而不是彼此覆盖。

**Non-Goals:**

- 不改变合法 GFM 表格的识别规则。
- 不重新设计 `Enter` / `Tab` 在已激活表格内部的现有扩行语义。
- 不引入新的公开 props、配置项或用户可见设置。
- 不扩展到非终端表格的整表删除手势，也不在本次引入新的整表工具栏按钮。

## Decisions

### 1. 表格激活阈值保持不变，但激活副作用要补齐首个空数据行

**决策**：继续沿用“合法 GFM 表格即激活结构化态”的现有阈值，不推迟激活时机；但当这次激活来自手动输入，且当前表格只有表头行与 delimiter 行时，编辑器必须立即补出首个空数据行。

**为什么这样做：**

- 用户已经输入到合法表格阈值，说明“我正在写表格”这个意图已经足够明确。
- 如果把激活时机继续往后推，只是把问题从“激活太早”变成“何时才算真正开始编辑表格”。
- 自动补首个空数据行后，用户可以连续录入第一行内容，不需要自己再把光标挪回表格。

**备选方案：**

- 延迟到用户输入第一条数据行后才激活。放弃，因为会让结构化态与源码态的切换阈值更难预测。
- 激活时不补行，只把光标放回表头。放弃，因为这仍然不能解决“继续录第一行”的核心痛点。

### 2. 文档末尾表格激活时，首个空数据行与尾随空行必须视为同一次 authoring continuity 处理

**决策**：当手动录入的表格在文档末尾第一次激活时，编辑器需要在一次连续处理里同时保证两件事：

- 表格内部有可继续录入的第一条空数据行
- 表格下方有一个普通文本空行作为出口

实现上必须被编排为同一个 transaction 内的连续 spec，而不是拆成独立的 follow-up transaction；语义上它们属于同一条 authoring continuity 规则。

**为什么这样做：**

- 这两条行为服务的是不同目标：前者保证“继续写表格”，后者保证“可以离开表格继续写正文”。
- 它们不是互斥关系，而是终端表格 authoring flow 的两个组成部分。
- 如果只保留其中一条，用户不是会卡在表格内部，就是会在文档末尾失去普通文本出口。

**备选方案：**

- 只保留首个空数据行，不保留尾随空行。放弃，因为终端表格依然没有稳定出口。
- 只保留尾随空行，不补首个空数据行。放弃，因为用户激活后仍然要手动回到表格内部才能继续录第一行。

### 3. 激活后的源码补写必须在 transaction filter 内与触发字符合成同一个事务

**决策**：手动录入触发表格首次激活时，自动补首个空数据行和尾随空行的源码改写，必须在 transaction filter 中与触发临界点的最后一个输入字符合成为同一个 transaction，而不能依赖 `ViewPlugin.update()` 之后的独立 follow-up transaction。

**为什么这样做：**

- CodeMirror 不允许在视图更新进行中再次同步 `dispatch`。
- 只有把“最后一个触发字符”和“系统补写”放进同一个 transaction，第一次撤销才会直接回到触发阈值之前。
- transaction filter 仍然允许在识别到首次激活后，把原始输入和后续规范化改写编排为单次事务内的连续 spec。

**备选方案：**

- 在 `update()` 内直接 `dispatch`。放弃，因为会触发 `Calls to EditorView.update are not allowed while an update is in progress`。
- 用独立 follow-up transaction 做系统补写。放弃，因为这会把首次激活拆成两个 undo step，无法满足本次 change 的撤销语义。

### 4. “手动输入”必须被定义成逐字符跨阈值，而不是任何形成合法表格的来源

**决策**：本次 authoring continuity 规则只应用于下面这类路径：

- 初始状态不是合法结构化表格
- 用户通过逐字符文本输入把源码推进到合法 GFM 表格阈值
- 这次输入直接导致结构化激活

完整表格的粘贴、撤销重放、程序化 `modelValue` 写入，或本来已经是合法表格的再次编辑，都不属于这条规则。

**为什么这样做：**

- 用户手动录入表格时，系统补首个空数据行和尾随空行是在延续写作意图。
- 如果把粘贴完整表格也纳入同一条规则，系统会在用户已经给出完整结构后再擅自补内容，显得过度主动。
- 实现上也更容易通过 transaction `userEvent` 与“前一状态不是表格、后一状态首次成为表格”的组合来判断。

**备选方案：**

- 任何形成合法表格的来源都触发补齐。放弃，因为会误伤粘贴与程序化写入。

### 5. 首次激活的系统补写必须与触发表格成立的最后一次输入共享同一个撤销语义

**决策**：当用户输入最后一个字符触发表格首次激活，而系统又自动补了首个空数据行或尾随空行时，这几步必须被视为同一个 undo boundary。用户第一次撤销时，应直接回退到“触发临界点之前”的源码状态。

**为什么这样做：**

- 从用户心智看，系统补写不是独立操作，而是最后一个输入字符引发的连带结果。
- 如果第一次撤销只撤掉系统补写，用户会看到表格一会儿结构化、一会儿半结构化，历史栈体验会非常别扭。
- 这也是避免“撤销一次后，再输入一个字符却触发不了同样激活”的关键。

**备选方案：**

- 把系统补写作为独立 undo step。放弃，因为这会破坏写作连续性。

### 6. 尾随空行删除守卫需要一个显式的“整表选中”状态，而不是复用当前 row / column 结构选中

**决策**：为终端表格删除守卫引入显式的“整表选中”状态，用于表达“用户第一次删除并不是要删掉空行，而是在确认是否删除整张表”。

**为什么这样做：**

- 现有 `structureSelection` 只覆盖整行或整列，不足以表达“整张表待删除”。
- 用户希望第一次删除先看到一个明确的选中结果，再按一次才真正删除，这需要一个可见且可测试的中间态。
- 如果继续复用某个 cell active 或 row/column selected 态，用户很难理解自己当前是否已经进入“删整表确认”。

**备选方案：**

- 第一次删除时直接把光标放进表格最后一个 cell。放弃，因为这仍然会把用户带回表格内部，而不是明确表达整表删除意图。
- 第一次删除时什么都不做，只弹 toast。放弃，因为这会把一个编辑流问题变成提示文案问题。

### 7. 整表选中必须是可退出的暂态，并保留一个锚点光标位置

**决策**：整表选中不是新的长期编辑 mode，而是终端删除守卫的暂态。进入整表选中时，系统必须保留一个锚点光标位置，即“触发守卫前那个文档末尾空行的行首”。当用户不再继续删除时：

- 鼠标点击空白处或其它部位，整表选中退出，点击按普通点击语义生效
- 方向键按该锚点位置的普通移动语义执行，并退出整表选中
- 文本输入按该锚点位置插入，并退出整表选中

**为什么这样做：**

- 这样用户不会被困在一个没有退出路径的删除确认态里。
- 方向键和继续输入都能回到“我本来就在表格下方空行”这条心智模型，而不是被强行带回表格内部。

**备选方案：**

- 整表选中期间冻结一切普通导航和输入。放弃，因为这会让状态显得像模态对话框而不是编辑器暂态。

### 8. 两段式删除守卫只拦截“文档最后一行 + 空行行首 + 上方为终端表格”的后退删除

**决策**：删除守卫严格限制在下面这个条件组合：

- 当前光标为空选区
- 当前行是文档最后一行
- 当前行为空行
- 光标位于该空行行首
- 紧邻上方的是终端结构化表格
- 用户执行的是后退删除（`Backspace` / 平台等价 Delete）

只有满足这一组条件时，第一次删除才接管为整表选中。

**为什么这样做：**

- 这个守卫是为了避免删除连接换行导致的表格语法坍塌，不应该扩散成普通删除行为的全局改写。
- 条件越精确，越能避免误伤其它末尾空行或普通段落删除行为。

**备选方案：**

- 任何位于表格下方空行的删除都拦截。放弃，因为会让普通删除行为变得过度保守。
- 拦截 `Backspace` 和 `Delete` 的所有方向删除。放弃，因为本问题真正危险的是删除前一字符所导致的换行合并。

### 9. 第二次删除整表后的收口结果必须固定，且撤销需要完整恢复表格与出口

**决策**：当用户在整表选中态下再次执行后退删除，编辑器删除整张表后，光标应停留在该表格上方一行。若用户随后撤销，整张表及其下方普通文本空行都必须完整恢复，且光标回到该尾随空行的行首。

**为什么这样做：**

- 删除后的光标落点如果不固定，用户会失去对“我删掉了什么、现在还在文档什么位置”的感知。
- 撤销必须恢复表格和尾随空行一起恢复，否则删除守卫本身会破坏原本要保护的 authoring continuity。

**备选方案：**

- 删除后把光标放到文档末尾任意位置。放弃，因为这会让收口结果不可预测。
- 撤销只恢复表格，不恢复尾随空行。放弃，因为会重新引入终端表格无出口的问题。

## Risks / Trade-offs

- [风险] 自动补首个空数据行会比用户显式输入更早地改写 Markdown 源码。  
  → 缓解：只在“手动输入首次激活且当前没有任何数据行”时触发，避免影响已存在的数据表格。

- [风险] 终端表格首次激活时同时补数据行和尾随空行，可能导致 selection / focus / scroll 的竞态。  
  → 缓解：把系统补写收敛到单个 composed transaction，并用自动化测试固定光标最终落点。

- [风险] 把系统补写并入同一个 undo boundary，会增加 transaction 编排复杂度。  
  → 缓解：在 design 和 spec 中先固定用户语义，再通过专门测试覆盖“最后一个触发字符 -> 首次撤销回到阈值前”的路径。

- [风险] 引入整表选中状态会扩大 `structuredTable` 的 UI 状态面。  
  → 缓解：把它限制为终端表格删除守卫专用，不扩展成新的通用结构操作模式。

- [风险] 两段式删除会让“我只是想删个空行”的路径多按一次键。  
  → 缓解：仅在最容易破坏表格语法的终端空行场景启用，换取结构安全与连续编辑体验。

## Migration Plan

- 这是纯前端交互语义调整，不涉及数据迁移或持久化格式升级。
- 发布时需要在变更说明中明确：`livePreview=true` 下，手动录入 Markdown 表格的激活后行为将更接近“继续表格写作”而非停留在源码草稿态。
- 如果实现出现回归，回滚策略是撤销本次 authoring-flow 相关的 transaction filter 编排和删除守卫逻辑，不影响既有表格源码格式。
